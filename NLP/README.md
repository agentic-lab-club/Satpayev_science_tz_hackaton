# TZ Analysis MVP

Анализатор и генератор улучшенных технических заданий (ТЗ) для научных и инновационных проектов программно-целевого финансирования (ПЦФ НИР). Вся логика реализована в формате интерактивного Jupyter Notebook (`tz_analysis_mvp.ipynb`).

## Основные возможности

- **Умный парсинг**: загрузка файлов форматов `.txt`, `.docx`, `.pdf` (с автоматическим OCR через `pytesseract` при необходимости).
- **Сверка со структурой ПЦФ**: автоматическое обнаружение обязательных разделов (Общие сведения, Цель, Задачи, Сроки, KPI, Бюджет, Ожидаемые результаты и др.).
- **Смысловой анализ**: выявление размытых формулировок, противоречий и неполных данных.
- **Explainable Score (0-100)**: оценка качества документа с прозрачной шкалой (полнота, структура, требования, KPI, конечные результаты).
- **Авто-улучшение текста**: генерация улучшенной версии (`improved_tz`) с добавлением отсутствовавших разделов.
- **Извлечение сущностей (NER)**: выделение сроков, параметров бюджета, KPI и требований.
- **Human-in-the-Loop**: интерактивное подтверждение/редактирование сущностей до финального сохранения (отключаемо).
- **Автономность**: поддержка работы через API (OpenRouter) с фоллбэком на эвристический режим без LLM (regex + правила).

## Быстрый старт

### 1. Установка зависимостей (Python 3.10+)
```bash
pip install -U python-docx pdfplumber pytesseract requests pydantic
```
*(Для работы OCR со сканами PDF в системе должен быть установлен Tesseract).*

### 2. Настройка LLM (Опционально)
По умолчанию пайплайн использует легковесные эвристики. Для максимального качества анализа укажите ключ OpenRouter:
```bash
export OPENROUTER_API_KEY="sk-or-v1-..."
# Опционально: смена модели (по умолчанию openai/gpt-oss-120b:free)
export OPENROUTER_MODEL="anthropic/claude-3-opus"
```

### 3. Запуск анализа
Запустите `tz_analysis_mvp.ipynb` в Jupyter Lab, Jupyter Notebook или VS Code.
По умолчанию notebook создаст и обработает `demo_tz_sample.txt`.

Чтобы проанализировать свой файл, измените переменную в финальной ячейке (раздел "Запуск"):
```python
FILE_PATH = "my_document.docx"
INTERACTIVE_CONFIRMATION = True # Или False для автоматического прогона
```

## Выходные артефакты

По результатам анализа формируется единый JSON-отчет в папке `notebook_outputs/tz_analysis_report.json`:
- `structure`: результаты структурного анализа (найденные/пропущенные/пустые разделы).
- `semantic`: извлеченные сущности, сроки, требования.
- `score`: распределение баллов и текстовое объяснение оценки.
- `recommendations`: конкретные шаги для доработки ТЗ.
- `improved_tz`: сгенерированный исправленный текст ТЗ.
- `confirmed_entities`: подтвержденные пользователем данные (при включенном interactive-режиме).

## Структура папки NLP

- `tz_analysis_mvp.ipynb` — Единый интерактивный pipeline продукта.
- `build_tz_analysis_notebook.py` — Скрипт-генератор ноутбука из исходников (если используется).
- `demo_tz_sample.txt` — Автогенерируемый тестовый файл.
- `notebook_outputs/` — Каталог для сохранения финальных JSON-отчетов (`tz_analysis_report.json`).
- `Tests.ipynb` — Дополнительный ноутбук для изолированных проверок.

## Формат входных и выходных данных (Input & Output)

### Инпут (Входной документ)
На входе пайплайн принимает файл с текстом технического задания. 
Поддерживаемые форматы:
* **TXT** — обычный текст (кодировка UTF-8).
* **DOCX** — документы Microsoft Word (извлекается текст из абзацев и таблиц).
* **PDF** — текстовые и сканированные отчёты (если PDF без текстового слоя, применяется OCR через Tesseract).

Параметр для передачи файла (в `tz_analysis_mvp.ipynb`):
```python
FILE_PATH = "my_document.docx"
```

### Аутпут (Выходной отчет)
Пайплайн возвращает Python dict, который затем сериализуется в JSON-файл в папку `notebook_outputs/tz_analysis_report.json`. 

Пример структуры выходного JSON-отчета:

    "weak_sections": ["задачи"],

        "section": "2. Цель",

      }
    ],
    "contradictions": [],
    "missing_elements": []

Также в `analysis_report["generate_improved_tz"]` можно получить структуру с улучшенной версией:
```json
{
  "summary_of_changes": ["..."],
  "improved_text": "Полный переписанный текст"
}
```

### Описание полей выходного JSON-отчета

* `file_path` (строка): Путь к проанализированному файлу.
* `raw_text_preview` (строка): Первые символы текста, извлеченного из документа.
* `structure` (объект): Результаты структурного парсинга документа.
  * `sections`: Список выделенных структурных блоков (заголовок, содержание, длина).
  * `found_sections`: Список обязательных разделов ПЦФ, которые удалось найти.
  * `missing_sections`: Обязательные разделы, которые полностью отсутствуют.
  * `weak_sections`: Разделы, которые найдены, но являются слишком короткими или не содержат достаточной информации.
  * `empty_sections`: Разделы, где есть заголовок, но нет текста.
  * `matched_required_sections`: Словарь соответствия эталонных названий разделов (например, "общие сведения") реальным заголовкам из текста.
* `semantic` (объект): Результаты семантического (смыслового) анализа.
  * `ambiguities`: Найденные размытые и неизмеримые формулировки (включает саму цитату, пояснение проблемы и рекомендацию по исправлению).
  * `contradictions`: Противоречия между требованиями/сроками внутри текста.
  * `missing_elements`: Детализированные ошибки отсутствия конкретики (например, отсутствие численных показателей или дат).
  * Массивы `requirements`, `deadlines`, `kpis`, `expected_results`: Предварительно извлеченные сущности.
* `score` (объект): Оценка качества по модели Explainable Score.
  * `total_score`: Итоговый балл (от 0 до 100).
  * `breakdown`: Распределение баллов по категориям (структура, полнота, требования, целостность, KPI, результаты).
  * `explanation`: Список текстовых объяснений того, как сформировалась оценка.
* `recommendations` (массив строк): Дедуплицированный список конкретных шагов по улучшению документа.
* `generate_improved_tz` / `improved_tz` (объект): Сгенерированный улучшенный вариант документа.
  * `summary_of_changes`: Список действий, выполненных при улучшении.
  * `improved_text`: Полная улучшенная версия текста (с добавленными шаблонами для пустых секций, уточнениями и исправлениями).
* `extracted_entities` (объект): Итоговый набор извлеченных сущностей сгруппированных по типам. Каждая сущность включает значение (`value`), раздел (`section`) и уровень уверенности модели в извлечении (`confidence` от 0.0 до 1.0).
* `confirmed_entities` (объект): Набор сущностей, которые были подтверждены (прошли проверку human-in-the-loop) либо автоматически подтверждены по высокому порогу `confidence`. Содержит те же категории: требования, сроки, KPI, результаты.
