# Система интеллектуального аудита технических заданий (MVP)

Бэкенд-сервис для автоматизированной оценки научно-исследовательских проектов и технических заданий по критериям «Научная новизна» и «Социально-экономический эффект».

---

## Архитектура системы

### Общая схема потока данных

```
┌─────────────┐     ┌──────────────┐     ┌──────────────┐     ┌─────────────┐     ┌──────────┐
│   txt.txt   │────▶│  JSON Parser │────▶│  RAG Engine  │────▶│  Gemini LLM │────▶│ Pydantic │
│  (JSON вход)│     │  (секции ТЗ) │     │  (фильтр)    │     │  (оценка)   │     │ (валидац)│
└─────────────┘     └──────────────┘     └──────────────┘     └─────────────┘     └────┬─────┘
                                                                                       │
                                                                                       ▼
                                                                                 ┌──────────┐
                                                                                 │  SQLite  │
                                                                                 │  (логи)  │
                                                                                 └──────────┘
```

### Этап 1: Входные данные — JSON из `txt.txt`

Система принимает структурированный JSON-документ через файл `txt.txt`. Формат входного файла:

```json
{
  "structure": {
    "sections": [
      {
        "title": "Название раздела",
        "content": "Полный текст раздела...",
        "length": 123
      }
    ]
  }
}
```

При вызове эндпоинта `/api/v1/evaluate_json` сервер:
1. Читает файл `txt.txt` из директории проекта.
2. Парсит JSON и извлекает массив `structure.sections`.
3. Склеивает все секции в единый текст, сохраняя заголовки как маркеры `## Название`.
4. Автоматически исправляет невалидный JSON (отсутствие внешних скобок `{}`, висячие запятые).

### Этап 2: RAG Pipeline (Retrieval-Augmented Generation)

> **Зачем нужен RAG?** Научные ТЗ могут содержать десятки страниц. Отправлять весь текст в LLM — дорого, медленно, и модель теряет фокус. RAG сжимает документ, оставляя только абзацы, релевантные критериям оценки.

RAG-конвейер реализован в модуле `rag_engine.py` и активируется только для документов **больше 10 000 символов**. Для коротких текстов этап пропускается.

#### 2.1 Чанкинг (сегментация текста)

```
Длинный текст (50 000 символов)
        │
        ▼
┌───────────────────────────────┐
│     chunk_text()              │
│                               │
│  chunk_size = 1500 символов   │
│  overlap    = 300  символов   │
│                               │
│  • Разрезает текст на блоки   │
│  • Перекрытие 300 символов    │
│    между соседними чанками    │
│    чтобы не терять контекст   │
│  • Разрыв на пробеле, а не   │
│    посреди слова              │
└───────────┬───────────────────┘
            │
            ▼
   ~33 чанка по 1500 символов
```

**Почему перекрытие (overlap)?** Если важная мысль оказалась на границе двух чанков, overlap гарантирует, что она попадёт хотя бы в один из них целиком.

#### 2.2 Векторизация и индексация (ChromaDB)

```
~33 чанка
    │
    ▼
┌─────────────────────────────────────────┐
│         ChromaDB (in-memory)            │
│                                         │
│  Embedding model: all-MiniLM-L6-v2      │
│  (Sentence-Transformers, ~80MB)         │
│                                         │
│  Каждый чанк → вектор 384 измерений     │
│  Все векторы сохраняются во временную   │
│  коллекцию с уникальным UUID            │
└─────────────────────────────────────────┘
```

Модель `all-MiniLM-L6-v2` — компактная нейросеть для семантических эмбеддингов. Она понимает смысл текста на уровне предложений и абзацев, а не просто совпадение слов.

#### 2.3 Семантический поиск (Query)

Система формирует **два целевых запроса эксперта**, каждый из которых описывает, что именно ищем в документе:

| Запрос | Текст запроса | Что ищет |
|--------|--------------|----------|
| **Научная новизна** | `научная новизна, методы исследования, инновации, актуальность проблемы, алгоритмы, сравнение с аналогами, разработанные схемы` | Разделы с описанием методов, алгоритмов, отличий от аналогов |
| **Социально-экономический эффект** | `социально-экономический эффект, бюджет, затраты, окупаемость, сэкономленное время, рабочие места, KZT, тенге, рентабельность, коммерциализация` | Разделы с метриками, бюджетами, KPI |

Для каждого запроса ChromaDB возвращает **Топ-7 самых похожих по смыслу чанков** (косинусное сходство векторов).

```
Запрос "научная новизна"  ──▶  Топ-7 чанков
Запрос "соц-эконом эффект" ──▶  Топ-7 чанков
                                    │
                                    ▼
                          Дедупликация (set)
                                    │
                                    ▼
                         Итого: до 14 уникальных
                         чанков (~21 000 символов)
```

#### 2.4 Результат RAG

```
Входной документ:     50 000 символов (20+ страниц)
После RAG фильтрации: ~21 000 символов (только суть)
Сжатие:               ~2.5x
```

Временная коллекция ChromaDB удаляется сразу после поиска, чтобы не засорять память сервера.

### Этап 3: LLM Оценка (Gemini API)

Отфильтрованный текст (или полный, если он короткий) отправляется в модель **Gemini 2.5 Flash** через OpenAI-совместимый SDK.

#### Системный промпт

Модель получает роль жёсткого технического аудитора с набором правил:

| Правило | Описание |
|---------|----------|
| **Анти-спам фильтр** | Если текст — не научный документ (рецепт, стихи, спам), модель выставляет `is_scientific_document = false` и 0 баллов |
| **Научная новизна** | Макс. 15 баллов. Штраф до 5/15 если только общие слова без конкретных методов |
| **Соц-экономический эффект** | Макс. 10 баллов. Штраф до 3/10 если нет точных метрик (KZT, %, штуки) |
| **Запрет фантазий** | Модель не имеет права придумывать цифры и метрики за автора |

#### Формат вызова

```python
client.chat.completions.create(
    model="gemini-2.5-flash",
    messages=[system_prompt, user_message],
    response_format={"type": "json_object"},  # Принудительный JSON
    temperature=0.0                            # Максимальная детерминированность
)
```

### Этап 4: Валидация и постобработка (Pydantic)

Сырой JSON-ответ от LLM проходит через строгую Pydantic-модель `TZEvaluationResponse`:

```python
class TZEvaluationResponse(BaseModel):
    is_scientific_document: bool    # Научный документ или спам?
    rejection_reason: str           # Причина отказа (если спам)
    scientific_novelty: CriterionEvaluation    # Детальная оценка новизны
    socio_economic_effect: CriterionEvaluation # Детальная оценка эффекта
    total_score: int                # Суммарный балл (вычисляется)
    max_total_score: int            # Максимум = 25
```

Каждый критерий (`CriterionEvaluation`) содержит:
- `score` / `max_score` — выставленный и максимальный баллы
- `extracted_text` — цитата из текста, подтверждающая оценку
- `found_features` — найденные конкретные элементы
- `missing_elements` — что отсутствует и нужно добавить
- `improvement_suggestion` — рекомендация по улучшению

**Постобработка** принудительно ограничивает баллы (новизна ≤ 15, эффект ≤ 10) и пересчитывает total.

### Этап 5: Логирование в SQLite

Каждый анализ сохраняется в таблицу `audit_logs`:

| Поле | Тип | Описание |
|------|-----|----------|
| `id` | UUID | Уникальный идентификатор |
| `created_at` | DateTime | Время анализа |
| `original_text` | Text | Исходный текст (первые 5000 символов) |
| `is_scientific_document` | String | `"true"` / `"false"` |
| `scientific_novelty_score` | Float | Балл за новизну |
| `socio_economic_score` | Float | Балл за эффект |
| `total_score` | Float | Суммарный балл |
| `raw_analysis_json` | JSON | Полный ответ LLM |

### Дополнительный модуль: Релевантность проектов (`relevance_scorer.py`)

Отдельный NLP-модуль для ранжирования списка проектов по актуальности. Два варианта скоринга:

| Вариант | Метод | Скорость | Точность |
|---------|-------|----------|----------|
| `score_by_embeddings()` | Sentence-Transformers (`all-MiniLM-L6-v2`) | ~2 сек | Высокая (семантическая) |
| `score_by_keywords()` | TF-IDF + cosine similarity | Мгновенно | Средняя (лексическая) |

## Развертывание проекта

1. Установка зависимостей:
   ```bash
   pip install -r requirements.txt
   ```
2. Конфигурация окружения — скопировать шаблон и вставить свой ключ:
   ```bash
   cp .env.example .env
   ```
   Отредактировать `.env`:
   ```env
   GEMINI_API_KEY="ваш_ключ_от_aistudio.google.com"
   GEMINI_MODEL="gemini-2.5-flash"
   ```
3. Подготовить входные данные — положить JSON в `txt.txt` (формат описан выше).
4. Запуск сервера:
   ```bash
   python tz_auditor.py
   ```
   Сервер: `http://localhost:8000`
   Swagger: `http://localhost:8000/docs`

## Документация API

### 1. Оценка ТЗ из JSON-файла
**Эндпоинт:** `POST /api/v1/evaluate_json`

Читает `txt.txt`, парсит JSON-секции, прогоняет через RAG + Gemini, возвращает оценку.

**Успешный ответ (200 OK):**
```json
{
  "is_scientific_document": true,
  "rejection_reason": "",
  "scientific_novelty": {
    "is_present": true,
    "score": 13,
    "max_score": 15,
    "extracted_text": "Предложена новая модель и схемы 1-4...",
    "found_features": ["схема 1", "сравнение моделей"],
    "missing_elements": ["Сравнительный анализ с метриками"],
    "improvement_suggestion": "Необходимо добавить конкретное сравнение."
  },
  "socio_economic_effect": {
    "is_present": true,
    "score": 2,
    "max_score": 10,
    "extracted_text": "облегчение понимания текста студентами",
    "found_features": ["улучшает процесс обучения"],
    "missing_elements": ["KZT", "Сэкономленные часы", "Рентабельность"],
    "improvement_suggestion": "Предоставьте конкретные метрики в часах или тенге."
  },
  "total_score": 15,
  "max_total_score": 25
}
```

### 2. Обработка сырого текста
**Эндпоинт:** `POST /api/v1/evaluate`
**Формат данных:** `application/json`

```json
{
  "tz_text": "Исходный текст технического задания..."
}
```
*Примечание: данный маршрут работает без механизма RAG и передает весь массив текста в LLM напрямую.*

### 3. Получение статистики
**Эндпоинт:** `GET /api/v1/statistics`

Возвращает агрегированные данные: количество проанализированных документов, отклонённый спам и средние баллы.
