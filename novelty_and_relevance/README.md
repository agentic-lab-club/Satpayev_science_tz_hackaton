# Система интеллектуального аудита технических заданий (MVP)

Бэкенд-сервис для автоматизированной оценки научно-исследовательских проектов и технических заданий по критериям «Научная новизна» и «Социально-экономический эффект».

## Архитектура и основные возможности

1. **LLM Интеграция**: Обработка данных осуществляется с помощью модели Gemini 2.5 Flash через официальный SDK с использованием жесткого системного промптинга.
2. **Строгая типизация (Pydantic)**: Входящие и исходящие данные валидируются. Система гарантирует возврат корректного JSON-ответа для взаимодействия с клиентской частью.
3. **RAG (Векторный поиск)**: Реализован механизм предварительной обработки длинных текстов (PDF/DOCX). Система сегментирует документ, генерирует эмбеддинги через модель `all-MiniLM-L6-v2` и передает в LLM только наиболее релевантные блоки (до 14 фрагментов). Это позволяет избежать превышения контекстного окна и снизить задержки.
4. **Валидация контента (Анти-спам)**: Встроен программный фильтр нерелевантных документов (кулинарные рецепты, художественная литература). При обнаружении нецелевого текста классификатор прерывает анализ.
5. **База данных и аналитика**: Все транзакции и результаты парсинга логируются в локальной СУБД (SQLite/SQLAlchemy) для последующего формирования метрик.

## Развертывание проекта

1. Установка зависимостей:
   ```bash
   pip install -r requirements.txt
   ```
2. Конфигурация окружения. В корневой директории (рядом с `tz_auditor.py`) необходимо создать файл `.env` со следующим содержимым:
   ```env
   GEMINI_API_KEY="ВАШ_КЛЮЧ_ДОСТУПА"
   GEMINI_MODEL="gemini-2.5-flash"
   ```
3. Запуск сервера:
   ```bash
   python tz_auditor.py
   ```
   Сервер будет доступен по адресу `http://localhost:8000`.
   Документация Swagger OpenAPI: `http://localhost:8000/docs`.

## Документация API

### 1. Обработка загружаемого файла (PDF, DOCX, TXT)
**Эндпоинт:** `POST /api/v1/evaluate_file`
**Формат данных:** `multipart/form-data`

| Ключ | Тип | Описание |
| --- | --- | --- |
| `file` | File | Текстовый документ или сканированный PDF (.pdf, .docx, .txt). |

**Успешный ответ (200 OK):**
```json
{
  "is_scientific_document": true,
  "rejection_reason": "",
  "scientific_novelty": {
    "is_present": true,
    "score": 13,
    "max_score": 15,
    "extracted_text": "Предложена новая модель и схемы 1-4...",
    "found_features": ["схема 1", "сравнение моделей"],
    "missing_elements": ["Сравнительный анализ с метриками"],
    "improvement_suggestion": "Необходимо добавить конкретное сравнение."
  },
  "socio_economic_effect": {
    "is_present": true,
    "score": 2,
    "max_score": 10,
    "extracted_text": "облегчение понимания текста студентами",
    "found_features": ["улучшает процесс обучения"],
    "missing_elements": ["KZT", "Сэкономленные часы", "Рентабельность"],
    "improvement_suggestion": "Предоставьте конкретные метрики в часах или тенге."
  },
  "total_score": 15,
  "max_total_score": 25
}
```

### 2. Обработка сырого текста
**Эндпоинт:** `POST /api/v1/evaluate`
**Формат данных:** `application/json`

```json
{
  "tz_text": "Исходный текст технического задания..."
}
```
*Примечание: данный маршрут работает без механизма RAG и передает весь массив текста в LLM напрямую.*

---

### 3. Получение статистики
**Эндпоинт:** `GET /api/v1/statistics`

Возвращает агрегированные данные из базы данных о количестве проанализированных документов и средних баллах.

---

### 4. Дополнительный модуль: Оценка релевантности (NLP)
Для сортировки списка проектов по актуальности или совпадению с трендами предусмотрен скрипт `relevance_scorer.py`. Он вычисляет семантическое косинусное сходство между проектами и заданным бенчмарком с использованием эмбеддингов локальной NLP-модели.
Запуск скрипта в режиме тестирования:
```bash
python relevance_scorer.py
```
