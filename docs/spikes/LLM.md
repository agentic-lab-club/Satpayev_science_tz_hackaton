# Сервис 1 — Input/Output с привязкой к шаблону программы

## Input сервиса 1

```json
{
  "document_id": "doc_001",
  "file_path": "data/input/tz.docx",
  "template_id": "science_program_v1"
}
```

`template_id` фиксирует, что документ проверяется по шаблону:
- `1.1`, `1.2`
- `2.1`, `2.2`
- `3` (с обязательным указанием конкретных пунктов стратегических/программных документов)
- `4.1`, `4.2`
- `5` (предельная сумма и разбивка по годам)

## Что делает сервис 1 внутри

### 1) Парсинг
Извлекает `raw_text`, метаданные, страницы, заголовки, нумерацию.

### 2) Нормализация структуры
Формирует `sections` с полями:
- `number`, `title`, `full_heading`, `parent_number`, `level`;
- `text`, `char_count`, `page_start`, `page_end`;
- `is_empty`, `is_short`.

### 3) Алгоритмическая проверка по шаблону
Проверяет:
- наличие обязательных разделов: `1`, `2`, `3`, `4`, `5`;
- наличие обязательных подпунктов: `1.1`, `1.2`, `2.1`, `2.2`, `4.1`, `4.2`;
- корректность нумерации и последовательности;
- пустые/слишком короткие разделы;
- разделы вне шаблона;
- для пункта `3`: есть ли конкретные номера/ссылки на документы;
- для пункта `4.1`: есть ли количественные показатели;
- для пункта `5`: есть ли сумма и разбивка по годам.

### 4) Смысловой анализ (LLM/NLP)
Ищет:
- противоречия между `2.*` (цели/задачи) и `4.*` (результаты);
- размытые формулировки без измеримости;
- отсутствие KPI и сроков;
- логические разрывы между задачами и итоговым эффектом;
- несоответствие между заявленным экономическим/экологическим/социальным эффектом и содержанием разделов.

LLM возвращает короткие объяснения в `brief_llm_explanation` для каждого `semantic_gap`.

## Как трактовать output в контексте этого шаблона

### `sections`
Главный нормализованный слой. По нему проверяется соответствие пунктам `1`–`5` и подпунктам `1.1`, `1.2`, `2.1`, `2.2`, `4.1`, `4.2`.

### `structure_overview`
Показывает, что отсутствует относительно шаблона, где сломана нумерация и какие пункты «вне формы».

### `algorithmic_gaps`
Формальные нарушения шаблона (без LLM):
- нет обязательного пункта/подпункта;
- пропуски нумерации;
- пустые/короткие разделы;
- нет обязательных численных показателей в `4.1`;
- нет обязательной бюджетной разбивки в `5`;
- в `3` нет конкретных ссылок на пункты стратегических документов.

### `semantic_gaps`
Смысловые проблемы (LLM/NLP):
- цели и задачи не переходят в результаты;
- «ожидаемые результаты» не подтверждают заявленный эффект;
- формулировки без измеримости/KPI;
- внутренние противоречия между разделами.

### `brief_llm_explanation`
Обязательно коротко, предметно и с привязкой к номерам пунктов.

### `llm_modification`
Результат генерации «идеального ответа» от LLM:
- `status` — статус генерации (`generated` / `failed`);
- `generation_goal` — цель переписывания;
- `ideal_response` — краткая сводка идеальной версии;
- `applied_changes` — список изменений по пунктам;
- `improved_document_text` — полный улучшенный вариант текста.

Пример хорошего брифа:

> 2.1 и 4.2 не согласованы: цель требует технологического результата, а в 4.2 описаны только общие рекомендации без измеримого эффекта и внедрения.
